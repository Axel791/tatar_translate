# Перевод «Достижений» лауреатов на татарский язык

⚠️ **Репозиторий должен быть ПРИВАТНЫМ** — в данных персональные сведения (ФИО, награды).

## Что это и где мы

Переводим колонку «Достижения» из `Пакет200.xlsx` (лист «Лауреаты», 14 651 строка) на литературный
татарский. Среди строк 14 301 уникальный текст (~6.4 млн знаков); переводим уникальные, дальше
переводы раскладываются по всем строкам автоматически (ключ = `sha1(текст)[:12]`).

**Статус на 2026-07-14:** переведён и принят пилот — 321 текст (Волна 1 + калибровочная порция),
в мастер-таблице заполнено 579 строк. Остаток — **13 980 текстов**, разбит на 4 равные части
(по ~3 500 текстов / ~1.58 млн знаков), по одной на ноутбук.

**Проверено A/B-тестами (не пересматривать без Тимофея):**
- Модель: **Sonnet** (Haiku дисквалифицирован — выдаёт псевдотатарские несуществующие слова).
- Уровень рассуждений: **max** (выбор Тимофея по качеству).
- Батч на агента: **25 текстов**. Параллельность: **3 воркфлоу × 8 агентов = 24 одновременно**.
- Стайлгайд `work/style_guide_dost.md` обязателен: наградная терминология, топонимы
  (Зеленодольск → Яшел Үзән), татаризация имён (Альфия → Әлфия), «января» → «гыйнвар».
- Алгоритмические «ускорители» проверены и отброшены: дедуп сегментов даёт 3.4%, шаблоны — 0.3%.
  Единственный рычаг — параллелизм агентов.

## Структура

```
Пакет200.xlsx                  мастер-таблица (финальную сборку делает только координатор)
work/style_guide_dost.md      стайлгайд перевода — читает каждый агент
work/pipeline.py              конвейер: status / slice / merge / apply / apply-part
work/dost_unique.jsonl        все уникальные тексты (для валидации merge)
work/dost_translations.jsonl  чекпоинт переводов (сид 321; растёт локально; НЕ коммитить до финала)
work/parts/part_N.jsonl       часть машины N — источник работы
work/parts/part_N.xlsx        та же часть таблицей (локальный контроль, apply-part)
work/queue/                   локальная нарезка батчей (в git не попадает)
work/batches/wave_*.jsonl     результаты агентов — ЕДИНСТВЕННОЕ, что машины пушат
```

## Быстрый старт на ноутбуке (человек)

1. `git clone git@github.com:Axel791/tatar_translate.git && cd tatar_translate`
2. `python3 -m venv .venv && .venv/bin/pip install openpyxl`
3. Узнай свой номер машины у Тимофея: **1, 2, 3 или 4** (часть = `work/parts/part_N.jsonl`).
4. Открой Claude Code в корне репозитория и скажи:
   > Прочитай README.md. Я машина №N. Работай по регламенту: запускай порции перевода моей
   > части, после каждой — merge, status, commit и push моих файлов. Продолжай до конца части.
5. Следи за прогрессом: `.venv/bin/python work/pipeline.py status N` и командой `/workflows`.

## Регламент для Claude (машина №N)

Цикл одной порции (≈600 текстов, ~15–25 мин, ~2.5M токенов агентов):

1. `git pull` (перед каждой порцией).
2. Нарезать порцию P (нумеруй порции 1, 2, 3… последовательно):
   `.venv/bin/python work/pipeline.py slice N P` → создаст до 24 файлов `work/queue/mNpP_b01..b24.json`.
   Повторный slice автоматически подхватывает брак прошлых merge (retry не нужен отдельно).
3. Запустить **три воркфлоу одним сообщением** (батчи b01–b08, b09–b16, b17–b24 соответственно)
   по шаблону ниже. Если slice дал меньше батчей (конец части) — раздели поровну.
4. Дождаться завершения всех трёх → `.venv/bin/python work/pipeline.py merge` →
   `.venv/bin/python work/pipeline.py status N`.
5. Зафиксировать результат:
   `git add work/batches/wave_mNpP_*.jsonl && git commit -m "m<N>: порция <P>, +<сколько> переводов" && git push`
   (при желании обновить свою таблицу: `pipeline.py apply-part N` — но её коммитить не обязательно).
6. Повторять, пока `status N` не покажет 100%. Затем финальный мини-slice подберёт остатки/ретраи;
   если slice выдал 0 батчей — часть закончена, сообщи Тимофею.

### Жёсткие правила

- **Коммитить только свои `work/batches/wave_mN*.jsonl`** (и при желании `part_N.xlsx`).
  НИКОГДА не коммитить: `dost_translations.jsonl`, `retry_keys.txt`, `work/queue/`. Никогда `git add -A`.
- Не трогать чужие части и мастер-`Пакет200.xlsx` (его собирает координатор в конце).
- Стайлгайд и промпт переводчика не менять без согласования с Тимофеем.
- Промпт агента и параметры (`model: 'sonnet'`, `effort: 'max'`, батч 25) — фиксированы.
- В Workflow-скриптах `args` приходит строкой — **списки батчей зашивать в скрипт литералом**.
- `Date.now()`/`Math.random()` в Workflow-скриптах недоступны.
- Если воркфлоу падает по лимитам подписки — пауза до сброса окна (обычно до 5 ч);
  чекпоинт ничего не теряет, продолжить тем же циклом.

### Шаблон воркфлоу (подставь: ROOT = абсолютный путь репо на этой машине, список батчей, номера)

```js
export const meta = {
  name: 'translate-mNpP-w1',
  description: 'Машина N, порция P, воркфлоу 1: батчи b01–b08, Sonnet max',
  phases: [{ title: 'Перевод' }],
}
const ROOT = '<АБСОЛЮТНЫЙ_ПУТЬ_К_РЕПО>'
const JOBS = ['mNpP_b01','mNpP_b02','mNpP_b03','mNpP_b04','mNpP_b05','mNpP_b06','mNpP_b07','mNpP_b08']
phase('Перевод')
const results = await pipeline(
  JOBS,
  name => agent(
`Ты — профессиональный переводчик с русского на литературный татарский язык.

1. Прочитай стайлгайд полностью: ${ROOT}/work/style_guide_dost.md — он ОБЯЗАТЕЛЕН: наградная терминология (Почетная грамота → Мактау грамотасы и т.д.), топонимы Татарстана (Казань → Казан, Зеленодольск → Яшел Үзән...), татаризация имён (Альфия → Әлфия, Кабировна → Кәбировна; неочевидные формы не изобретать), изафет, порядок слов, даты («января» → «гыйнвар»), аббревиатуры (ООО → ҖЧҖ...), краевые случаи.
2. Прочитай входной файл: ${ROOT}/work/queue/${name}.json — JSON-массив объектов {"k": "<ключ>", "ru": "<русский текст>"}.
3. Переведи КАЖДЫЙ текст массива. Требования:
   - Смысловой литературный перевод, не пословный. Ничего не выдумывать и ничего не терять по содержанию.
   - Обрезанный на полуслове хвост исходника — опустить огрызок, НЕ досочинять.
   - Все числа, даты, проценты, номера сохранить точно как в исходнике.
   - Русские имена и все инициалы не менять; фамилии — в написании исходника.
   - Кавычки «ёлочки», сбалансированные; вложенные — “лапки”.
   - Мусорные исходники без смысла («О», «-», «.») — скопировать как есть.
   - Каждый перевод — ОДНА строка без \\n.
4. Запиши результат через Write в файл: ${ROOT}/work/batches/wave_${name}.jsonl — ровно столько строк, сколько объектов во входном массиве; каждая строка — валидный JSON {"k": "<тот же ключ>", "tt": "<перевод>"}; UTF-8, кириллица как есть (не \\u-эскейпы).
5. В финальном сообщении верни ТОЛЬКО: «OK <число переведённых>» и до 3 сомнительных ключей с причиной одной строкой.

Работай автономно до конца, вопросов не задавай.`,
    { label: name, phase: 'Перевод', model: 'sonnet', effort: 'max', agentType: 'general-purpose' }
  )
)
const ok = results.filter(Boolean).length
log('Готово агентов: ' + ok + '/' + JOBS.length)
return { done: ok, total: JOBS.length }
```

Второй и третий воркфлоу — то же самое с b09–b16 и b17–b24 (и своими name/description).

## Контроль качества

`merge` пускает в чекпоинт только пары, прошедшие автопроверки: сохранность чисел/дат, длина
(защита от обрезания), баланс кавычек «», отсутствие непереведённых русизмов-маркеров
(«награжден», «работает», «является»…). Брак не теряется: ключи уходят в `retry_keys.txt`,
и следующий `slice` той части снова выдаст эти тексты агентам. По опыту брак ~0.5–1%.

Выборочная человеческая проверка: открой `part_N.xlsx` после `apply-part N` и просмотри
случайные 20–30 пар. Эталон качества — файл Волны 1 у Тимофея.

## Финальная сборка (только координатор — Тимофей, машина 1)

1. `git pull` — собрать волны всех машин.
2. `.venv/bin/python work/pipeline.py merge` — всё в один чекпоинт (дубли отсеются сами).
3. Остатки/брак: `slice N 99` по каждой части → добить мини-порциями.
4. `.venv/bin/python work/pipeline.py apply` — заполнить лист «Перевод достижений» в `Пакет200.xlsx`.
5. Финальный коммит: чекпоинт + `Пакет200.xlsx` → push. Готовый файл — `Пакет200.xlsx`.
